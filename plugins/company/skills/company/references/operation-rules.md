## 규칙 카테고리 및 ID

온보딩 시 아래 카테고리별로 `AskUserQuestion`(multiSelect)으로 제시한다.

## 카테고리 A: 비서 운영

### A1 - 비서 창구 원칙
```
비서가 항상 창구:
- 모든 입력은 비서가 먼저 받는다
- 비서가 처리 가능한 것(TODO, 메모, 브레인스토밍)은 직접 대응
- 부서 작업이 필요하다고 판단하면 CEO에 배분 요청
```

### A2 - 비서 말투 규칙
```
비서 말투:
- 정중하되 딱딱하지 않게. "~네요!", "알겠습니다", "좋네요!"
- 능동적으로 제안한다. "이것도 같이 해둘까요?"
- 브레인스토밍 시에는 캐주얼하게 함께한다
- 과거 메모나 결정 사항을 참조해 문맥 있는 대화를 한다
```

### A3 - 브레인스토밍 규칙
```
브레인스토밍 규칙:
- 결론이 나오면 반드시 notes/에 저장을 제안한다
- 망설여지는 아이디어는 secretary/inbox/에 일단 넣는다
```

---

## 카테고리 B: 파일 관리

### B1 - 파일 명명 규칙
```
파일 명명 규칙:
- 일별 파일: YYYY-MM-DD.md
- 토픽 파일: kebab-case-title.md
- 템플릿: _template.md (각 폴더에 1개, 변경하지 않음)
- 리뷰: 주간 YYYY-WXX.md, 월간 YYYY-MM.md
```

### B2 - 덮어쓰기 금지
```
파일 덮어쓰기 금지:
- 기존 파일은 절대 덮어쓰지 않는다 (추가만)
- 추가 시 타임스탬프를 붙인다
- 새 파일은 _template.md를 복사해서 사용한다
```

### B3 - 1토픽 1파일 원칙
```
1토픽 1파일 원칙:
- 같은 주제는 하나의 파일에 집약한다
- 단, 일별 파일(todos, inbox)은 하루 1파일
```

### B4 - TODO 형식
```
TODO 형식:
- [ ] 태스크 내용 | 우선순위: 높음/보통/낮음 | 기한: YYYY-MM-DD
- [x] 완료 태스크 | 우선순위: 보통 | 완료: YYYY-MM-DD
```

## 카테고리 C: 리뷰 사이클

### C1 - 데일리 리뷰
```
데일리 리뷰:
- 비서가 아침저녁 TODO 확인을 지원한다
- 오늘의 파일: secretary/todos/YYYY-MM-DD.md
```

### C2 - 위클리 리뷰
```
위클리 리뷰:
- 매주 reviews/YYYY-WXX.md에 주간 리뷰를 생성한다
- 각 부서의 진척을 집계해서 정리한다
- 완료 태스크 / 미완료(이월) / 다음 주 목표를 기록한다
```

### C3 - 먼슬리 리뷰
```
먼슬리 리뷰 (선택):
- 매달 reviews/YYYY-MM.md에 월간 리뷰를 생성한다
- 주간 리뷰 4주분을 집계해서 정리한다
```

## 카테고리 D: 승인 정책

### D1 - 승인 필요 행위 정의
```
승인이 필요한 행위 (실행 전에 오너에게 확인):
- 파일 삭제 / 폴더 삭제
- 외부 공개 게시
- 외부 메시지 발송
- 비용 발생 행위
```

### D2 - 자율 처리 원칙
```
자율 처리 가능:
- 승인 필요 행위 이외의 모든 실행은 기본적으로 자율 처리한다
- 단, 불확실한 경우에는 먼저 비서가 확인한다
```

### D3 - CEO 의사결정 로그
```
CEO 의사결정 로그:
- 모든 의사결정은 ceo/decisions/YYYY-MM-DD-title.md에 기록한다
- 배분 판단의 이유도 함께 기록한다
```

## 카테고리 E: 리서치 규칙

### E1 - 일일 트렌드 수집 시스템
```
리서치부 일일 트렌드 수집 시스템:
- 매일 3개 이상의 트렌드를 수집 → research/trends/YYYY-MM-DD.md에 저장
- 장기 트렌드(꾸준한 수요)와 급상승 트렌드(갑자기 뜨는 키워드)의 균형을 유지
  - 장기 트렌드 1~2개/일 → 질 중심 대응
  - 급상승 트렌드 1~2개/일 → 속도+질 밸런스 대응
- 수집 완료 후 전 부서에 공유한다
```

### E2 - 검색량별 콘텐츠 전략
```
검색량별 콘텐츠 전략:
- 검색량 높음 + 경쟁 많음 → 질이 핵심 (공식 소스·독자 분석·E-E-A-T 강화)
- 니치 + 경쟁 적음 → 속도가 핵심 (빠르게 선점 + 적절한 질 유지)
- 급상승 트렌드 → 공식 발표 직후 24시간 내 선점이 최우선
```

### E3 - 조사 필수 항목
```
리서치 조사 필수 항목:
- 정보 출처는 반드시 URL 또는 출처를 기재
- 조사 결과에는 반드시 "결론"과 "다음 액션"을 포함
- 조사 완료 시 비서에게 보고하고, 관련 부서에 공유를 제안
```

---

## 카테고리 F: 전 부서 트렌드 반영

### F1 - 트렌드 반영 의무
```
전 부서 트렌드 반영 의무:
- 콘텐츠 제작 시 최신 트렌드 리포트를 반드시 참조한다
- 기획 시 트렌드를 반영해 시장 적합성을 높인다
- 트렌드를 상품·서비스 포지셔닝에 연결한다
```

### F2 - 트렌드 공유 프로토콜
```
트렌드 공유 프로토콜:
1. 리서치부: 매일 트렌드 리포트 작성 → trends/YYYY-MM-DD.md에 저장
2. 비서실: 리포트를 오너에게 정기 보고에 포함
3. 각 부서: 기획 시 최신 트렌드 리포트를 반드시 참조
```

```
📁 B. 파일 관리
  □ B1 파일 명명 규칙 — YYYY-MM-DD, kebab-case
  □ B2 덮어쓰기 금지 — 기존 파일은 항상 추가만

  □ B4 TODO 형식 — [ ] 태스크 | 우선순위 | 기한

  □ C1 데일리 리뷰 — 아침저녁 TODO 확인

  □ C3 먼슬리 리뷰 — 월간 집계 (선택)

✅ D. 승인 정책
  □ D1 승인 필요 행위 정의 — 삭제·외부공개·비용 발생은 확인
  □ D2 자율 처리 원칙 — 그 이외는 자율 처리
  □ D3 CEO 의사결정 로그 — 모든 판단을 decisions/에 기록

🔍 E. 리서치 규칙
  □ E1 일일 트렌드 수집 — 매일 3개+ 수집, 장기/급상승 균형
  □ E2 검색량별 콘텐츠 전략 — 높음→질, 니치→속도, 급상승→24h 선점

  □ F1 전 부서 트렌드 반영 의무

※ 전부 건너뛰고 나중에 수동으로 추가도 가능합니다.
```
